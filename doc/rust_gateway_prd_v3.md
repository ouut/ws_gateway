# Rust 智能网关 —— 核心需求文档 v3（Agent 可执行版）

> 本版基于 v2 调整：**明确不做鉴权（P3）、不做完整性校验（P4）**，按内网可信环境简化实现。其余协议扩展性、背压策略、可观测性设计保持 v2 不变。

---

## 0. 变更说明（相对 v2）

| 编号 | v2 设计 | v3 调整 |
|---|---|---|
| P3 | WS 连接需携带 Token 鉴权 | **移除鉴权**，任何客户端携带 `room`/`user` 参数即可连接，视为内网可信环境 |
| P4 | Payload 末尾追加 CRC16，可配置开启校验 | **移除 CRC 字段**，网关不做数据完整性校验，Payload 出错由业务方自行兜底 |
| 其余 | Version/Length/Sequence 头部字段、心跳超时、背压丢帧策略、可观测性、客户端封包规范 | 保持不变 |

> ⚠️ 部署提示：由于取消了鉴权与完整性校验，本网关**必须部署在完全可信的内网/局域网环境**，不得直接暴露 WS(8080) 或 UDP(9999) 端口到公网。如未来需要公网接入，需重新引入 P3/P4 机制。

---

## 1. 系统定位与核心价值

单文件、零外部业务依赖、单机高性能二进制数据总线。

- **万物皆用户**：体感设备、网页前端、AI 预测节点，在网关眼中都是平等的 Client。
- **全链路二进制**：网关、AI、前端间用定长二进制信封头 + 二进制 Payload 通讯。
- **业务弱侵入**：网关只解析固定长度的头部字段用于路由，不解析、不校验 Payload 内容。
- 系统仍需要长期的可观测性、日志与协议演进能力（见第 8 章），"开发完成后永不维护"不作为设计前提。

## 3. 连接与身份管理

### 3.1 接入方式
- WS 接入：`ws://host:8080/ws?room=<ROOM>&user=<USER>`（无需 token）
- UDP 接入：设备在应用层自行封装头部后发送数据报，无需预先"连接"，头部携带 Room/User 用于路由（见第 6 章）。

### 3.2 身份声明（不鉴权，仅路由用途）
- `room`/`user` 参数仅用于网关内部路由，**不做任何身份校验**，客户端可自由声明任意值。
- 网关信任所有接入方，这是明确的设计取舍（内网可信环境），不在网关层面做安全加固。

### 3.3 重复连接处理
- 同一 `RoomID + UserID` 组合重复连接时：**后连接者踢掉先连接者**（避免僵尸连接占位），并记录一条 WARN 日志。

### 3.4 心跳与超时清理
- WS 层使用标准 Ping/Pong，网关每 10s 发一次 Ping，30s 未收到 Pong 则视为断线，主动关闭并从连接池移除。
- 移除后检查所在 Room 是否还有活跃连接，若无则销毁该 Room 的字典条目。
- UDP 层不维护"连接"概念，不做超时判断，由业务方（AI/前端）基于 Sequence 号自行判断设备是否离线。

## 4. 二进制协议规范 v3（22 字节头部）

```
0        1        2        3     4       10      18                 22
+--------+--------+--------+--------+.......+--------+--------+--------+
|Version | PktType|TargetTy| Reserv |RoomID(6B)|UserID(8B)| Sequence(4B) |Length(2B)|
+--------+--------+--------+--------+.......+--------+--------+--------+
```

| 字段 | 偏移 | 长度 | 说明 |
|---|---|---|---|
| Version | 0 | 1B | 协议版本号，当前为 `0x02`。网关收到不识别的版本直接丢弃并记录日志。 |
| Packet Type | 1 | 1B | `0x01` RAW_MOTION / `0x02` AI_EVENT / `0x03` SYSTEM_CMD / `0x04` HEARTBEAT |
| Target Type | 2 | 1B | `0x01` BROADCAST / `0x02` UNICAST |
| Reserved | 3 | 1B | 保留字节，供未来扩展（如优先级/QoS），当前必须为 `0x00` |
| Room ID | 4 | 6B | ASCII，不足右侧补 `\0`。**禁止使用多字节 UTF-8 字符**，网关侧对非 ASCII 输入直接拒绝连接。 |
| User ID | 10 | 8B | 同上，ASCII 定长 |
| Sequence | 18 | 4B | 无符号整数，发送方单调递增。用于接收方判断丢包/乱序，网关不做任何处理，仅透传。 |
| Length | 22 | 2B | Payload 字节长度（不含头部）。网关据此校验实际收到的字节数是否匹配，不匹配则丢弃该包并计入错误 metrics。 |

> 头部共 24 字节（含 Sequence/Length），保留了协议可诊断性和可扩展性，属于合理权衡，不影响"微秒级路由"目标（仍是定长解析，O(1)）。**不再包含 CRC 字段**，Payload 完整性完全由业务方自行负责，网关不做任何校验、不因数据损坏而丢包（除非 Length 不匹配）。

### 4.1 Payload 规范
- `0x01` RAW_MOTION：固定 2652 字节 ARKit 浮点数组。
- `0x02` AI_EVENT：`[1 Byte Action ID]`。
- `0x03` SYSTEM_CMD：预留，格式由业务方自定义，网关不解析。
- `0x04` HEARTBEAT：Payload 为空，Length = 0，仅用于应用层心跳（区别于 WS Ping/Pong，供 UDP 场景使用）。

## 5. 路由核心与性能要求

### 5.1 并发数据结构
- `DashMap<RoomID, DashMap<UserID, mpsc::Sender<Bytes>>>`，分段锁，不同房间无锁竞争。

### 5.2 零拷贝实现方式
- UDP/WS 读到的原始字节，立即转换为 `bytes::Bytes`（Arc 引用计数的不可变缓冲区）。
- 广播场景下，对多个订阅者的分发只是 `Bytes::clone()`（增加引用计数，非内存拷贝），而不是逐个 `Vec<u8>::clone()`。
- 验收标准：压测时，广播 N 个连接的内存分配次数应与 N 无关（仅头部路由查找是 O(N)，Payload 内存零拷贝）。

### 5.3 背压与丢帧策略
- 每个连接的发送 channel 使用**有界队列**（建议容量 64~128）。
- 队列满时的策略按 Packet Type 区分：
  - `RAW_MOTION`（实时性优先，允许丢帧）：`try_send` 失败则**丢弃最老的一帧**，插入最新帧（保证低延迟优先于完整性）。
  - `SYSTEM_CMD`（可靠性优先）：`try_send` 失败则记录 WARN 日志，可选择短暂 `await` 重试（不超过 50ms），仍失败则断开该连接（视为客户端消费能力异常）。
- 此策略必须写清楚在配置文件中可调（不同业务场景权衡不同）。

### 5.4 性能指标
- 路由延迟（收包到转发完成）：P99 < 500μs（单机，1000 并发连接）。
- UDP 接收器：单独协程死循环 `recv_from`，收到后立即 `tokio::spawn` 路由任务，接收循环自身不做任何阻塞操作。
- 需提供压测脚本 / 报告作为验收物料（见第 11 章）。

## 6. 客户端封包规范

- 任何向网关发送数据的一方，**必须自行按第 4 章协议组装 24 字节头部**，网关不提供任何"代封包"的便利层。
- 建议提供参考实现：
  - Python（AI 预测端）：`struct.pack('>BBBB6s8sIH', ...)` 组装头部的示例。
  - Unity/C#（体感设备，如适用）：`BinaryWriter` 示例。
- Sequence 号由发送方自行维护（每个 UserID 独立计数器，从 0 开始单调递增，允许回绕）。
- 接收方（AI/前端）如果需要处理丢包/乱序，自行基于 Sequence 判断，网关不提供任何保证，也不做数据校验。

---

## 7. 静态文件服务

- 托管 `public/` 目录到 HTTP 根路径。
- 静态文件 I/O 使用独立的 tokio task / 线程池（如 `tokio::fs` + 有界并发），避免拖慢核心路由的 worker 线程调度。生产环境建议改用 nginx/CDN 承担静态资源，网关内置服务仅作为开发期便利。

## 8. 可观测性与运维

### 8.1 日志
- 使用 `tracing` crate，区分 INFO（连接建立/断开）、WARN（丢帧、心跳超时、重复连接踢出）、ERROR（内部异常）。

### 8.2 Metrics（建议 `:9090/metrics`，Prometheus 格式）
- `active_connections_total`（按 room 打标签）
- `active_rooms_total`
- `packets_routed_total`（按 Packet Type/Target Type 打标签）
- `packets_dropped_total`（区分丢弃原因：队列满 / Length 不匹配 / 版本不识别）
- `route_latency_seconds`（直方图）

### 8.3 优雅关闭
- 监听 `SIGTERM`/`SIGINT`，收到后停止接受新连接，等待现有队列排空（或超时后强制关闭），再退出进程。

## 9. 边界情况与异常处理清单（Agent 实现时必须覆盖）

- [ ] 收到的数据包总长度 < 24 字节 → 直接丢弃，计入错误 metric，不 panic。
- [ ] Length 字段与实际收到字节数不符 → 丢弃。
- [ ] Room ID / User ID 含非 ASCII 字节 → WS 连接阶段拒绝；UDP 包直接丢弃。
- [ ] 目标 RoomID 不存在（房间已销毁）→ 静默丢弃 + DEBUG 日志，不报错。
- [ ] UNICAST 目标 UserID 不在房间内 → 静默丢弃 + DEBUG 日志。
- [ ] 同 Room+User 重复连接 → 踢掉旧连接，见 3.3。
- [ ] 发送 channel 满 → 按 5.3 策略处理，不得阻塞路由核心。
- [ ] 进程收到 SIGTERM → 优雅关闭，见 8.3。
- [ ] 未知 Packet Type / Version → 丢弃 + 计入 metric，不影响其他连接。

## 10. 非目标（Out of Scope）

- 不做分布式/多机部署。
- 不做消息持久化/回放。
- 不做跨房间广播。
- **不做鉴权体系**（明确设计取舍，仅限内网可信环境使用）。
- **不做数据完整性校验**（明确设计取舍，Payload 损坏由业务方自行兜底）。
- UDP/WS 端口不做任何公网安全加固，严禁暴露到公网。

---

## 11. 验收标准（Definition of Done）

1. 单机 1000 WS 连接、100 房间压测下，P99 路由延迟 < 500μs，无内存持续增长（用 `heaptrack` 或类似工具验证无泄漏）。
2. Kill -9 模拟异常断线后，30s 内连接池正确清理僵尸连接（心跳超时机制生效）。
3. 队列满场景下丢帧策略符合 5.3 描述，不出现无界内存增长。
4. 提供最小可运行示例：一个 Python 脚本模拟 AI 客户端、一个 HTML 页面模拟前端，跑通"广播 + 单播"两条路径。
5. `/metrics` 端点可用，包含第 8.2 节列出的所有指标。
6. 边界情况清单（第 9 章）全部有对应单元测试或集成测试覆盖。
7. 部署文档中明确注明"仅限内网使用，不做鉴权和数据校验"的安全边界说明。

## 12. 技术栈建议

- 异步运行时：`tokio`
- WS + HTTP：`axum`（同时支持路由复用、WS upgrade、静态文件服务）
- 并发字典：`dashmap`
- 零拷贝缓冲：`bytes`
- 日志：`tracing` + `tracing-subscriber`
- Metrics：`metrics` + `metrics-exporter-prometheus`
